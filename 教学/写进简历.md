# 智扫通 — 扫地机器人智能客服 Agent 系统

## 一句话概述

基于 LangChain 1.x + ReAct 推理范式 + RAG 检索增强，从零构建的垂直领域智能客服系统，覆盖售前咨询、故障排查、个性化使用报告生成三大场景，支持流式对话与动态提示词切换。

---

## 项目描述（建议用于简历"项目经历"栏）

**项目背景：** 扫地机器人品类存在「售前选购决策复杂、售后使用问题专业、用户使用数据难以洞察」三大痛点，传统关键词客服无法理解复杂意图。本项目构建了一个具备自主思考与工具调用能力的 AI Agent 智能客服。

**核心工作：**

1. **Agent 推理引擎搭建**：基于 LangChain 1.x 新版 `create_agent` API 构建 ReAct（思考→行动→观察）Agent，Agent 能自主分析用户意图、决策工具调用链路、整合多源信息生成回答，替代传统硬编码路由逻辑。

2. **Middleware 中间件机制（亮点）**：利用 LangGraph Runtime 上下文传递能力，实现 3 类横切关注点的解耦——
   - `@wrap_tool_call`：工具调用全生命周期监控（日志记录、异常捕获、运行时上下文注入）
   - `@before_model`：模型调用前消息状态可视化，便于多轮推理调试
   - `@dynamic_prompt`：**基于 Runtime Context 动态切换系统提示词**，无侵入实现「客服模式」与「报告生成模式」的无缝切换，避免 Agent 主流程臃肿

3. **完整 RAG 检索链路**：
   - 文档加载层：支持 PDF / TXT 双格式，PyPDFLoader + TextLoader 统一入口
   - 去重机制：4KB 流式分块计算文件 MD5，持久化记录避免重复入库
   - 智能分片：`RecursiveCharacterTextSplitter` 按中文标点递归切分（`\n\n` → `\n` → 。！？ → 空格），chunk_size=200 / overlap=20 保证语义完整
   - 向量检索：Chroma 向量库持久化 + DashScope text-embedding-v4，Top-K 检索后经 LLM 总结生成最终回答

4. **多工具编排**：封装 7 个垂直场景工具（RAG 知识检索、天气查询、用户定位、用户 ID 获取、月份获取、外部数据检索、报告上下文触发器），Agent 在 ReAct 循环中自主编排调用顺序，支撑「深圳最近适合用扫地机器人吗」这类需要多工具协作的复杂查询。

5. **工程化实践**：
   - 工厂模式封装模型实例化（ChatModel / Embeddings），支持平滑切换模型厂商
   - YAML 配置驱动全部参数（模型、分片策略、提示词路径、向量库），零硬编码
   - 统一日志体系（按日期分文件 + 控制台双输出 + DEBUG/INFO 分级）
   - 流式输出：`agent.stream(stream_mode="values")` 实现 token 级流式响应，Streamlit `write_stream` 逐字符渲染

**技术栈：** Python 3.11 / LangChain 1.3 / LangGraph 1.2 / Chroma 1.5 / Streamlit 1.58 / 通义千问 qwen3.7-max / DashScope text-embedding-v4 / YAML 配置驱动

**项目成果：**
- 实现 3 大核心客服场景的智能应答：售前 RAG 问答、环境适配咨询、个性化使用报告生成
- Middleware 动态提示词切换方案使 Agent 主流程代码零侵入，新场景接入仅需新增提示词文件 + 工具
- 完整 RAG 链路支持 PDF/TXT 双格式知识库，MD5 去重机制保证知识库迭代的幂等性

## STAR 法则面试表述（用于面试口述）

**S (Situation)：** 扫地机器人客服场景中，用户问题横跨售前选购、售后故障、使用报告三类，传统 QA 机器人无法理解"深圳最近适合用扫地机器人吗"这类需要多步推理 + 多数据源的复杂意图。

**T (Task)：** 构建一个能自主思考、决策工具调用、整合多源信息的 AI Agent 智能客服系统。

**A (Action)：**
- 选用 LangChain 1.x 新版 Agent API，基于 ReAct 范式构建思考-行动-观察循环
- 设计 Middleware 中间件层，将工具监控、日志、动态提示词切换等横切关注点从 Agent 主流程中解耦
- 利用 LangGraph Runtime Context 实现基于上下文的提示词动态切换——Agent 调用特定工具后，中间件自动切换系统提示词为"报告生成模式"
- 搭建完整 RAG 链路，MD5 去重 + 中文智能分片保证检索质量
- 封装 7 个工具，Agent 在多轮 ReAct 循环中自主编排调用顺序

**R (Result)：** 系统支持售前咨询、环境适配、报告生成三大场景智能应答。Middleware 方案使提示词切换零侵入 Agent 主流程，扩展新场景仅需新增提示词文件 + 工具注册。

## 面试可能追问 & 准备要点

| 追问方向 | 准备要点 |
|---|---|
| 为什么用 Middleware 而不是 if/else？ | 横切关注点解耦，Agent 主流程不感知提示词切换逻辑；符合开闭原则，新场景零侵入 |
| ReAct 推理流程是怎样的？ | Thought → Action → Observation → Thought 循环，Agent 自主判断信息是否充足 |
| RAG 检索质量如何保证？ | 中文标点递归分片保证语义块完整；MD5 去重避免重复文档；Top-K=3 控制上下文长度 |
| 动态提示词切换的实现原理？ | `@dynamic_prompt` 装饰器 + LangGraph Runtime.context，工具调用时注入标记，下轮模型调用前检测并切换 |
| 如果切换模型厂商怎么办？ | 工厂模式封装，新增 Factory 子类即可，调用方无感知 |
| 向量库为什么选 Chroma？ | 轻量嵌入式、原生持久化、适合中小规模（百万级 chunk 以下），开发验证阶段成本低 |
