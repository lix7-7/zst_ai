# 智扫通项目 · 面试介绍指南

---

## 一、开场白（30 秒电梯演讲）

> 我做了一个基于 LangChain 的智能客服 Agent 系统，面向扫地机器人这个垂直领域。它不是一个传统的 QA 机器人，而是一个能**自主思考、自己决定调用什么工具、自己判断信息够不够**的 AI Agent。
>
> 技术上最大的亮点是我用 LangChain 1.x 的 Middleware 机制实现了一套**动态提示词切换方案**——Agent 在对话过程中识别到用户要生成报告时，能自动把系统提示词从"客服模式"换成"报告写手模式"，整个过程 Agent 主流程一行代码都不用改。

---

## 二、3 分钟详细介绍

### 第一步：先讲业务问题（10 秒）

> 扫地机器人这个品类有三个客服痛点：售前不知道怎么选、售后出了问题不会排查、用户也不知道自己的机器人用得怎么样。传统关键词匹配的客服根本处理不了"深圳最近回南天适合用扫地机吗"这种需要多步推理的问题。

### 第二步：讲技术方案（60 秒）

> 所以我基于 LangChain 1.x 的 `create_agent` API 搭了一个 ReAct Agent。ReAct 就是「思考→行动→观察→再思考」的循环——Agent 收到问题后先想"我现有信息够不够"，不够就决定调哪个工具，拿到结果后再判断"现在够了吗"，不够继续调，够了就生成回答。
>
> 这个 Agent 挂了 7 个工具：知识库 RAG 检索、天气查询、用户定位、用户 ID 获取、月份获取、外部数据检索，还有一个用来触发模式切换的标记工具。
>
> 整个系统的架构是分层的：Streamlit 做对话 UI → Agent 编排层做推理和工具调度 → RAG 检索层做知识库问答 → 模型层做工厂模式封装。

### 第三步：讲核心亮点——Middleware（60 秒）

> 我想重点讲一下 Middleware 这个设计，这是我觉得最有意思的部分。
>
> 问题背景是这样的：用户有两种典型诉求——一种是普通的客服问答，另一种是"帮我生成这个月的使用报告"。这两种场景需要的系统提示词完全不一样，客服模式要求 Agent 反复思考、谨慎回答，报告模式要求 Agent 直接按流程取数、生成 Markdown。
>
> 如果我用 if/else 在主流程里判断，Agent 代码会越来越臃肿，加一个场景就要改核心逻辑。所以我的方案是利用 LangGraph 的 Runtime Context——一个可以在 Agent 运行过程中携带的上下文对象。
>
> 具体做法：
> 1. 我定义了一个 `fill_context_for_report` 工具，它本身什么都不做，就是个触发器
> 2. 当 Agent 判断用户要做报告时，会调用这个工具
> 3. 我的 `monitor_tool` 中间件拦截到这次调用，在 Runtime Context 里打一个 `report=True` 的标记
> 4. 下轮模型调用前，`@dynamic_prompt` 中间件检测到这个标记，自动把系统提示词从 `main_prompt.txt` 换成 `report_prompt.txt`
>
> 这样做的好处是：Agent 主流程完全不知道提示词切换的存在，它是纯粹的"推理+调工具"。以后要加新的对话模式，我只需要写一个新的提示词文件和一个新的标记工具，核心代码零改动。

### 第四步：讲 RAG 链路（30 秒）

> RAG 这部分我搭了完整链路：支持 PDF 和 TXT 两种格式的文档加载，用文件 MD5 做增量去重避免重复入库，分片用了 `RecursiveCharacterTextSplitter` 按中文标点做递归切分保证语义完整，向量库用 Chroma 做持久化，检索 Top-3 后交给 LLM 总结。

### 第五步：收尾（10 秒）

> 整个项目虽然数据层目前还是 mock 的，但 Agent 的推理链路、Middleware 架构、RAG 管线都是完整可跑的，后续接入真实 API 只需要替换工具的实现，架构不用动。

## 三、面试官追问 · 标准答案

### Q1：你为什么选 LangChain 而不是自己写？

> 两个原因。第一，LangChain 1.x 的 `create_agent` 已经帮我们处理了 ReAct 循环、工具调用的序列化/反序列化、ToolMessage 的注入这些底层细节，自己写的维护成本太高。第二，我看重它的 Middleware 体系——`@wrap_tool_call`、`@before_model`、`@dynamic_prompt` 这些 hook 给了我一个标准化的横切关注点扩展机制，这比我自己在代码里到处插 hook 要干净得多。

### Q2：动态提示词切换还有没有别的实现方式？

> 当然有，但各有利弊：
>
> - **在主流程里 if/else**：最简单，但 Agent 代码会随场景增长而膨胀，违反开闭原则。
> - **拆成多个 Agent 实例**：售前一个 Agent、报告一个 Agent，前端路由。好处是隔离彻底，坏处是如果用户先咨询再要报告，会话状态要跨 Agent 传递，增加复杂度。
> - **Middleware + Runtime Context（我的方案）**：好处是无侵入、易扩展，坏处是依赖 LangGraph 的 Runtime 机制，有一定框架绑定。
>
> 选哪种取决于场景复杂度。当前场景只有两种模式，Middleware 方案是最优的——如果未来有 10 种模式，我可能会考虑拆 Agent 或引入状态机。

### Q3：RAG 检索不准怎么办？

> 我分层回答：
>
> - **分片层面**：chunk_size=200 + overlap=20 + 中文标点分隔符，保证语义块完整。如果发现检索结果割裂，可以调大 chunk_size 或加父文档召回。
> - **检索层面**：当前是简单的 Top-K 语义检索，如果准确率不够，可以上混合检索（语义 + BM25 关键词）、Rerank 模型做精排、或者 Multi-Query 检索（同义改写多次检索再合并）。
> - **质量评估层面**：目前系统缺少评估闭环，后续应该建测试集，统计召回率和答案准确率，这条路线上我还写了 TODO。

### Q4：你这里 7 个工具 Agent 不会选错吗？

> 有可能，尤其在工具描述不够清晰或者用户问题模糊的时候。我的防御措施：
>
> 1. **工具描述写得详细**：每个工具的 `@tool(description=...)` 我都写了核心能力、使用场景、调用规则，这部分其实就是在做 prompt engineering。
> 2. **系统提示词约束**：我在 `main_prompt.txt` 里写了强约束规则——比如报告场景必须先调 `fill_context_for_report`，没有调这个工具禁止调 `fetch_external_data`。
> 3. **最多 5 次工具调用兜底**：防止 Agent 陷入死循环，5 次后直接回复"不知道"。
>
> 如果要进一步提升，可以加一个 Tool Selection Router（用一个小模型先判断意图再分发给对应工具组），或者用 LangSmith 追踪工具调用链路做离线分析。

### Q5：为什么用 Chroma 而不是 Milvus / Pinecone / Elasticsearch？

> Chroma 是轻量级嵌入式向量库，部署零成本，非常适合开发和验证阶段。当前知识库就 6 个文档，chunk 加起来几百条，Chroma 完全够用。如果未来数据量到百万级，需要分布式和高可用，那时候再迁到 Milvus 或 Elasticsearch。工厂模式让这种迁移只影响 VectorStore 这一个类，上层 RAG Service 和 Agent 工具都不需要改。

### Q6：工厂模式在这个项目里是不是过度设计了？

> 这个项目当前只有一家模型厂商（阿里 DashScope），看起来用工厂模式是有点重。但我是有意为之——第一，我写这个项目的时候就想清楚了未来可能要切 OpenAI 或本地模型，工厂模式把切换成本压到了零。第二，LangChain 本身就对不同厂商的 ChatModel 做了统一抽象，我的工厂只是在这之上加了一层配置驱动的语法糖，代价很低。

### Q7：你提到流式输出，具体是怎么实现的？

> 两条线：
>
> - **Agent 端**：`create_agent` 返回的 agent 支持 `.stream()` 方法，我用 `stream_mode="values"` 拿到每次状态更新后的完整消息列表，取最后一条消息的 `content` 做 yield。这不是严格的 token 级流式，因为 LangChain Agent 的流式粒度是"每次 LLM 调用或工具调用完成"。
> - **前端端**：Streamlit 的 `write_stream` 接收一个 generator，我用一个 `capture` 函数包装了一下，在逐字符 yield 的同时把完整内容缓存到列表里，这样既能实现打字机效果，又能在 rerun 时恢复完整历史。
>
> 如果要做到真正的 token 级流式，需要用 `stream_mode="messages"` 或者直接调 LLM 的 stream API，但那样就绕过了 Agent 的工具调用逻辑，需要自己处理。

### Q8：项目还有什么不足？如果继续做你会做什么？

> 坦诚说有几个方向：
>
> 1. **真实 API 接入**：天气和用户数据现在都是 mock，接真实 API 后要考虑超时、重试、降级。
> 2. **多轮对话记忆管理**：目前没有做消息窗口截断和历史摘要，长对话会爆 token，需要加滑动窗口 + 摘要压缩。
> 3. **评测体系**：没有测试集，无法量化回答质量。需要构造 Ground Truth，统计召回率、准确率、幻觉率。
> 4. **可观测性**：接 LangSmith 或 LangFuse 做 Trace，否则出了问题很难定位是哪个环节。
> 5. **服务化部署**：目前是 Streamlit 单体，应该拆成 FastAPI + 前端，用 Docker Compose 编排。
>
> 这些我都在 README 的路线图里列了，优先级最高的是评测体系和真实 API 接入。

## 五、一句话总结（面试官问"你做的这个项目一句话怎么说"）

> 我用 LangChain 1.x 的 Agent API 加自研 Middleware 机制，从零搭了一个能自主推理、动态切换提示词的垂直领域智能客服系统，覆盖了 RAG 问答、多工具编排、流式对话的完整链路。
